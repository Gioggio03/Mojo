================================================================================
  MOSTREAM — Source-Rate Benchmark Report
  Image Processing Pipeline: Source -> Gray -> Blur -> Sharp -> Sink
  Image size: 512x512 | N=1000 immagini | Hardware: 24 core fisici
================================================================================


OBIETTIVO
---------
Valutare l'efficacia del parallelismo nella libreria MoStream misurando:
  1. Il throughput massimo teorico (velocita' del solo Source)
  2. Come il parallelismo degli stage di trasformazione si avvicina a quel limite
  3. Se il framework scala in modo efficiente all'aumentare dei worker


SETUP
-----
Pipeline testata:
  Source -> Grayscale -> GaussianBlur -> Sharpen -> Sink

Stage di trasformazione:
  - Grayscale:    converte RGB in scala di grigi (formula luminanza)
  - GaussianBlur: applica kernel 3x3 gaussiano (stage piu' costoso)
  - Sharpen:      applica kernel 3x3 di sharpening

Thread budget:
  - Source(1) + Sink(1) = 2 thread fissi
  - Massimo 22 thread per gli stage di trasformazione
  - Totale massimo: 24 thread = 24 core fisici (no hyperthreading)

Configurazioni testate:
  - Source baseline:  Source -> PassThrough -> Sink  (ceiling teorico)
  - Sequenziale:      tutti gli stage con 1 worker  (P=1)
  - Uniform P=2..7:   tutti gli stage con lo stesso numero di worker P
  - Smart configs:    worker proporzionali al costo di ogni stage
                      (piu' worker a Blur, meno a Gray)


RISULTATI
---------

Phase 0 — Source baseline (tetto teorico)
  Throughput: 1580 img/s
  Questo e' il limite fisico della pipeline: la velocita' con cui il
  Source riesce ad emettere immagini senza alcuna elaborazione.
  Nessuna configurazione puo' superarlo.

Phase 1 — Sequenziale (P=1 per ogni stage)
  Throughput: 73 img/s  (4.6% del source)
  Con un solo worker per stage la pipeline elabora 73 immagini al
  secondo. Il collo di bottiglia e' GaussianBlur, lo stage piu'
  costoso computazionalmente.

SPIEGAZIONE DELLO SPEEDUP SUPERLINEARE (efficienza > 100%)
-----------------------------------------------------------
Lo speedup con P worker per stage supera P (es. P=7 -> 8.1x invece
di 7x). Questo fenomeno e' chiamato "speedup superlineare" e non e'
un errore di misura: ha una spiegazione precisa nel contesto delle
pipeline con code.

La baseline P=1 NON e' un programma sequenziale puro. Anche con
un solo worker per stage, MoStream utilizza:
  - Thread separati per ogni stage
  - Code MPMC lock-free tra stage adiacenti
  - Operazioni atomiche per la sincronizzazione

Con P=1, ogni immagine deve attraversare un ciclo completo di
push/pop per ogni queue, e il singolo worker di ogni stage alterna
continuamente tra elaborazione e operazioni di sincronizzazione.
Questo overhead e' proporzionalmente alto rispetto al lavoro utile.

Con P=N worker per stage, mentre un worker elabora un'immagine,
un altro puo' eseguire le operazioni di queue. L'overhead di
sincronizzazione si distribuisce tra i worker e il tempo effettivo
per immagine scende al di sotto di T_elaborazione / N. Il risultato
e' uno speedup > N rispetto alla baseline P=1.

In sintesi: la baseline P=1 e' penalizzata dall'infrastruttura del
framework piu' di quanto lo sia P=N, quindi il rapporto non e'
puramente P ma P + recupero_overhead. Questo e' un comportamento
atteso e positivo: significa che il framework diventa piu' efficiente
all'aumentare del parallelismo.

ANALISI
-------

1. Il parallelismo funziona e supera la scalabilita' lineare
   Lo speedup con P worker per stage e' costantemente superiore a P:
   con P=7 si ottiene 8.1x. L'efficienza e' stabile tra il 115% e
   il 118% per tutti i valori di P testati. Come spiegato sopra,
   questo e' dovuto alla riduzione dell'overhead di sincronizzazione
   per immagine all'aumentare dei worker.

2. La scalabilita' e' perfettamente lineare e stabile
   L'efficienza rimane costante al 115-118% senza alcun degrado
   all'aumentare di P. Non si osserva contention sulle code MPMC,
   false sharing o overhead di scheduling fino a 23 thread simultanei.
   MoStream scala in modo ideale nel range testato.

3. La distribuzione smart batte quella uniforme a parita' di thread
   Concentrare i worker sugli stage piu' lenti e' sempre piu'
   efficiente:
     - G2 B8 S4   (16 thread): 533 img/s
     - Uniform P=5 (17 thread): 427 img/s
   Con un thread in meno, la distribuzione smart produce il 25% in
   piu'. Con il budget completo (24 thread), la config smart
   G2 B14 S6 raggiunge 808 img/s contro i 591 img/s della
   uniforme P=7 con 23 thread (+37%).

4. Il gap con il source e' strutturale
   Anche con tutti i 24 core fisici si arriva al 51% del source
   (808 vs 1580 img/s). La causa e' il costo computazionale degli
   stage di image processing, non l'overhead del framework. Per
   avvicinarsi al source rate servirebbero stage piu' efficienti
   (SIMD, tiling, accesso sequenziale alla memoria), non piu' worker.


CONCLUSIONE
-----------
MoStream scala in modo eccellente. La libreria non introduce overhead
significativo: l'efficienza parallela e' costantemente al 115-118%
per tutti i livelli di parallelismo testati (P=2..7, fino a 23 thread
simultanei). Lo speedup supera il valore ideale lineare perche' con
piu' worker per stage l'overhead di sincronizzazione delle code MPMC
si distribuisce, rendendo il framework piu' efficiente all'aumentare
del parallelismo.

Il limite delle prestazioni e' interamente determinato dal costo
computazionale degli stage di image processing, non dal framework.
